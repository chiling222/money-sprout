## 1. 產品與技術範圍

### 1.1 產品定位

面向台灣小資族年輕人的記帳與理財 App，核心訴求是「**讓存錢像養成遊戲**」。

### 1.2 功能範圍（v1）

| 編號 | 功能 | 狀態 |
|---|---|---|
| F1 | 快速記帳（支出／收入），可自訂常用分類 | 納入 v1 |
| F2 | 每月總預算（收入）＋分類預算，拉桿分配，使用達 85% 提醒 | 納入 v1 |
| F3 | 存錢目標（金額、目標日、每月需存、進度條、達成徽章） | 納入 v1 |
| F4 | 資產儀表板（淨資產＝現金＋儲蓄＋投資） | 納入 v1 |
| F4b | 投資／儲蓄以「資產搬家（轉帳）」處理，記一筆投資即自動由現金移轉，淨資產精準 | 納入 v1 |
| F5 | 首頁收支圓餅圖（圓心顯示本月結餘） | 納入 v1 |
| F6 | 多人分帳（手動輸入名字、墊款人、應付金額、付款狀態） | 納入 v1 |
| F7 | 「不明損失」分類，處理帳實不符的零頭差額 | 納入 v1 |
| F8 | 分帳「7 天未結清」**App 內**提醒 | 納入 v1 |
| F9 | 餘額校正（對帳：輸入實際餘額，自動補差額分錄） | v2 規劃 |
| F10 | 主動推播通知（手機通知列） | v2+ 進階 |
| F11 | 分帳跨帳號連動（朋友也是使用者，雙向同步） | 未排程 |

### 1.3 技術堆疊

- **前端**：原生 HTML / CSS / JavaScript（無建置流程），以 GitHub Pages 靜態託管。
- **驗證**：Firebase Authentication，Google 登入。
- **資料庫**：Cloud Firestore（NoSQL，文件／集合模型）。
- **Firebase SDK 引入方式**：靜態網站採用 CDN 版 SDK（ES module import），不需 npm 建置。

### 1.4 重要技術約束

- GitHub Pages 為純靜態託管，**無伺服器端程式碼**。任何需要「定時排程／伺服器主動推播」的功能（F10）都無法只靠前端完成，需另接 Cloud Functions + Cloud Messaging。
- Firebase 的 Web API Key 會出現在前端原始碼中，**這是正常且設計上可公開的**，安全性不靠隱藏金鑰，而是靠「**Authentication + Firestore Security Rules**」。仍建議於 Google Cloud Console 對該金鑰設定 HTTP referrer 限制，降低被濫用風險。

## 2. 核心設計原則

> 這一章是整份資料結構的地基，請務必先讀。

### 2.1 單一事實來源：交易（transactions）是唯一真相

所有「花了多少 / 收了多少」的數字，**一律由 `transactions` 彙總計算**，不另存一份會各自飄移的數字。

- 預算頁的「各分類已花金額」＝ 篩選該月、該分類的交易加總。
- 首頁圓餅圖、收入／支出／結餘 ＝ 由交易彙總。
- **好處**：永遠不會發生「總額和明細對不起來」的經典記帳 bug。

> **效能備註**：若日後交易量大、即時彙總變慢，可在 `budgets/{period}` 文件中存一份 `spentByCategory` 快取，於每次寫入交易時同步更新。但快取僅為加速用途，**交易仍為權威來源**，校驗時以交易為準。

### 2.2 金額一律以「整數新台幣元」儲存

TWD 日常記帳無角分需求，所有金額欄位為**正整數**，幣別預設 `TWD`。正負意義由 `type` 欄位決定，不在金額上放負號（避免彙總時符號錯亂）。

### 2.3 月份以 `period` 字串去正規化

每筆交易、每份預算都帶一個 `period`（格式 `YYYY-MM`，例：`2026-06`）。這讓「查某月所有交易」變成單欄位等值查詢，便宜又快，省去日期範圍查詢與索引成本。

### 2.4 使用者資料完全隔離

所有業務資料都掛在 `users/{uid}/...` 之下。安全規則保證「**使用者只能存取自己 uid 底下的資料**」。v1 分帳對象只是文字名字，不涉及跨帳號，因此不需要任何共享集合。

### 2.5 時間與時區

- 所有時間戳以 Firestore `Timestamp`（UTC）儲存。
- 顯示與「月份歸屬、7 天提醒」的計算，一律換算為 **Asia/Taipei (UTC+8)** 後再判斷。
- **帳務月份採日曆月**（每月 1 號起算），`settings.monthStartDay` 固定為 `1`。`period` 即該交易日所屬之日曆月。

### 2.6 投資／儲蓄是「資產搬家」，不是支出

把錢拿去投資或儲蓄，錢仍是使用者的，只是換個帳戶存放，**不應計入「支出」**，否則淨資產會失真（明明錢還在卻被當成花掉）。因此：

- **使用者操作維持單純**：在介面上仍是「記一筆投資 5,000 / 記一筆儲蓄 6,000」，無需理解任何術語。
- **系統內部以「轉帳（transfer）」處理**：記一筆投資 5,000 時，系統自動「現金 −5,000、投資 +5,000」。淨資產總額不變，僅內部分布改變。
- **轉帳不計入當月收支與圓餅圖**：它既非支出也非收入，純粹是資產在帳戶間移動。
- **預算頁的呈現**：投資、儲蓄仍可保留在預算頁作為「每月目標金額」格子（維持養成遊戲的存錢心態），但其達成是以「已轉入金額」計，而非「已花費」。

> 設計總結：**操作層對使用者無感（記一筆就好），記帳層精準（淨資產正確）**。這是同時兼顧易用與正確的關鍵設計。

## 3. Firestore 資料結構

### 3.1 集合總覽

```
users/{uid}                       使用者主檔（個人資料、設定、徽章）
  ├─ categories/{categoryId}      分類與收入來源（含自訂、含不明損失）
  ├─ budgets/{period}             每月預算（doc id = YYYY-MM）
  ├─ transactions/{txId}          每筆收支交易（單一事實來源）
  ├─ goals/{goalId}               存錢目標
  ├─ splits/{splitId}             多人分帳
  └─ accounts/{accountId}         資產帳戶（現金／儲蓄／投資）
```

> 採「使用者底下開子集合」的結構，是 Firestore 多租戶的標準做法：天然隔離、安全規則好寫、查詢不需到處帶 uid 條件。

---

### 3.2 `users/{uid}` — 使用者主檔

文件 ID 直接使用 Firebase Auth 的 `uid`。

| 欄位 | 型別 | 說明 | 範例 |
|---|---|---|---|
| `uid` | string | 同文件 ID，冗餘存放方便讀取 | `"a1B2c3..."` |
| `displayName` | string | 來自 Google 登入 | `"Chichi"` |
| `email` | string | 來自 Google 登入 | `"chichi@gmail.com"` |
| `photoURL` | string | Google 頭像網址 | `"https://..."` |
| `settings` | map | 使用者偏好設定（見下） | — |
| `badges` | map | 已解鎖徽章，key 為徽章 ID，值含解鎖時間 | 見下 |
| `createdAt` | timestamp | 帳號首次建立時間（伺服器時間） | — |
| `updatedAt` | timestamp | 最後更新時間 | — |

**`settings` 物件**：

| 子欄位 | 型別 | 預設 | 說明 |
|---|---|---|---|
| `currency` | string | `"TWD"` | 幣別 |
| `monthStartDay` | number | `1` | 每月帳務起算日（見第 7 章待確認） |
| `overspendAlertRatio` | number | `0.85` | 分類預算超支提醒門檻 |
| `splitReminderDays` | number | `7` | 分帳未結清提醒天數 |
| `theme` | string | `"sprout"` | 主題色 |

**`badges` 物件範例**：

```json
"badges": {
  "first_save":   { "earnedAt": "2026-06-01T10:00:00Z" },
  "invest_start": { "earnedAt": "2026-06-05T09:30:00Z" }
}
```

> 徽章「定義清單」（ID、名稱、圖示、解鎖條件）建議寫在**前端程式碼的常數表**，不存 Firestore，避免每次調整文案都要改資料。Firestore 只記「誰在何時解鎖了哪個 ID」。

### 3.3 `categories/{categoryId}` — 分類與收入來源

支出分類與收入來源共用此集合，以 `type` 區分。

| 欄位 | 型別 | 說明 | 範例 |
|---|---|---|---|
| `id` | string | 同文件 ID | `"food"` |
| `type` | string (enum) | `"expense"` \| `"income"` | `"expense"` |
| `name` | string | 顯示名稱 | `"日常餐飲"` |
| `emoji` | string | 圖示 | `"🍱"` |
| `color` | string | HEX 色碼（收入來源可省略） | `"#FF9F43"` |
| `isSystem` | boolean | 系統內建（true）或使用者自訂（false） | `true` |
| `special` | string \| null | 特殊用途標記：`"unknown_loss"`（不明損失）、`"unknown_gain"`（不明收入，F9 用） | `null` |
| `sortOrder` | number | 顯示排序 | `10` |
| `archived` | boolean | 是否封存（不刪除以保留歷史交易關聯） | `false` |
| `createdAt` | timestamp | — | — |
| `updatedAt` | timestamp | — | — |

**v1 系統內建支出分類**：日常餐飲、大餐、交通、娛樂、購物、訂閱、**不明損失（`special:"unknown_loss"`）**、投資、儲蓄。

**v1 系統內建收入來源**：薪水、獎金、副業、投資收益、紅包、退款、其他、**不明收入（`special:"unknown_gain"`，預設封存，供 F9 餘額校正使用）**。

> **設計重點**：分類採「封存 (archived) 而非刪除」。因為歷史交易以 `categoryId` 關聯分類；若硬刪分類，舊交易會變成孤兒。封存可讓它從新增介面消失，但歷史報表仍能正確顯示。

### 3.4 `budgets/{period}` — 每月預算

文件 ID 即月份字串 `YYYY-MM`，天然保證一個月只有一份預算。

| 欄位 | 型別 | 說明 | 範例 |
|---|---|---|---|
| `period` | string | 同文件 ID | `"2026-06"` |
| `income` | number | 當月總預算（總收入） | `35000` |
| `allocations` | map | 各分類預算金額，key=categoryId，value=金額 | 見下 |
| `createdAt` | timestamp | — | — |
| `updatedAt` | timestamp | — | — |

**`allocations` 範例**：

```json
"allocations": {
  "food": 6000, "feast": 2000, "trans": 1500, "fun": 2000,
  "shop": 3000, "sub": 800, "misc": 500, "invest": 5000, "save": 6000
}
```

> **不存 `spent`（已花金額）**：已花一律由 `transactions` 即時彙總（見 2.1）。`allocations` 只存「使用者拉桿設定的預算上限」。

### 3.5 `transactions/{txId}` — 交易（核心）

| 欄位 | 型別 | 說明 | 範例 |
|---|---|---|---|
| `id` | string | 同文件 ID（自動產生） | `"tx_8f3k"` |
| `type` | string (enum) | `"expense"` \| `"income"` \| `"transfer"` \| `"adjustment"` | `"expense"` |
| `amount` | number | 正整數金額 | `95` |
| `categoryId` | string \| null | 關聯分類（收入則為收入來源 ID；`transfer` 可為 null 或對應的投資／儲蓄目標格） | `"food"` |
| `accountId` | string | 影響的帳戶，預設 `"cash"`（`transfer` 不使用此欄，改用下方 from/to） | `"cash"` |
| `fromAccountId` | string \| null | **僅 `transfer` 使用**：轉出帳戶 | `"cash"` |
| `toAccountId` | string \| null | **僅 `transfer` 使用**：轉入帳戶 | `"invest"` |
| `note` | string | 備註（可空） | `"午餐 雞腿便當"` |
| `date` | timestamp | 交易發生日（使用者可改） | — |
| `period` | string | 由 `date` 換算的 `YYYY-MM`（日曆月，去正規化） | `"2026-06"` |
| `source` | string (enum) | `"manual"` \| `"reconciliation"`（餘額校正自動產生） | `"manual"` |
| `createdAt` | timestamp | 建立時間（伺服器） | — |
| `updatedAt` | timestamp | — | — |

**`type` 語意**：
- `expense`：支出，使現金（或指定帳戶）減少，計入當月收支與圓餅圖。
- `income`：收入，使現金增加，並計入當月 `income`。
- `transfer`：**資產搬家**（投資／儲蓄）。同時 `fromAccountId −amount`、`toAccountId +amount`。**不計入收支、不進圓餅圖**，僅改變淨資產內部分布。使用者操作層仍呈現為「記一筆投資／儲蓄」。
- `adjustment`：餘額校正（F9）自動產生的差額分錄，依差額正負，分類掛在「不明損失」或對應的「不明收入」。

**建議索引**（Firestore 複合索引）：
- `period (==) + date (desc)`：查某月交易、依日期排序。
- `period (==) + categoryId (==)`：查某月某分類加總。

### 3.6 `goals/{goalId}` — 存錢目標

| 欄位 | 型別 | 說明 | 範例 |
|---|---|---|---|
| `id` | string | 同文件 ID | `"goal_jp"` |
| `name` | string | 目標名稱 | `"日本賞楓"` |
| `emoji` | string | 圖示 | `"✈️"` |
| `targetAmount` | number | 目標金額 | `60000` |
| `savedAmount` | number | 已存金額 | `38000` |
| `targetDate` | string | 目標完成月份 `YYYY-MM` | `"2026-12"` |
| `status` | string (enum) | `"active"` \| `"completed"` \| `"archived"` | `"active"` |
| `completedAt` | timestamp \| null | 達標時間（觸發「完成目標」徽章） | `null` |
| `createdAt` | timestamp | — | — |
| `updatedAt` | timestamp | — | — |

**衍生欄位（前端即時計算，不入庫）**：
- 進度百分比 ＝ `savedAmount / targetAmount`
- 每月需存 ＝ `ceil((targetAmount − savedAmount) / 剩餘月數)`

> **選用擴充**：若日後想保留「每次存錢的歷史紀錄」，可加子集合 `goals/{goalId}/contributions/{cId}`，`savedAmount` 改由其加總。v1 直接存 `savedAmount` 即可。

### 3.7 `splits/{splitId}` — 多人分帳

| 欄位 | 型別 | 說明 | 範例 |
|---|---|---|---|
| `id` | string | 同文件 ID | `"split_kt"` |
| `title` | string | 分帳名稱 | `"墾丁三天兩夜"` |
| `emoji` | string | 圖示 | `"🏖️"` |
| `totalAmount` | number | 總金額 | `7200` |
| `payer` | string | 墊款人名字（手動文字，含 `"我"`） | `"我"` |
| `splitMethod` | string (enum) | `"even"`（平分）\| `"custom"`（自訂） | `"even"` |
| `members` | array&lt;map&gt; | 參與者明細（見下） | — |
| `status` | string (enum) | `"open"`（未結清）\| `"settled"`（已結清） | `"open"` |
| `date` | timestamp | 消費發生日 | — |
| `createdAt` | timestamp | **建立時間，作為 7 天提醒的計算基準** | — |
| `settledAt` | timestamp \| null | 全員付清的時間 | `null` |

**`members` 陣列元素**：

| 子欄位 | 型別 | 說明 |
|---|---|---|
| `name` | string | 參與者名字（手動輸入） |
| `share` | number | 應付金額 |
| `paid` | boolean | 是否已付款 / 已收款 |
| `paidAt` | timestamp \| null | 標記已付的時間 |

**範例文件**：

```json
{
  "title": "墾丁三天兩夜", "emoji": "🏖️",
  "totalAmount": 7200, "payer": "我", "splitMethod": "even",
  "status": "open",
  "members": [
    { "name": "我",   "share": 1800, "paid": true,  "paidAt": "2026-06-08T..." },
    { "name": "阿哲", "share": 1800, "paid": true,  "paidAt": "2026-06-10T..." },
    { "name": "小美", "share": 1800, "paid": false, "paidAt": null },
    { "name": "阿凱", "share": 1800, "paid": false, "paidAt": null }
  ],
  "date": "2026-06-08T...", "createdAt": "2026-06-08T...", "settledAt": null
}
```

> **`members` 用陣列的取捨**：陣列無法用查詢條件單獨更新某個元素，標記付款時需「讀整份 → 改 → 寫回」。對 v1（一筆分帳人數不多）完全足夠，且讀寫一份文件最單純。若日後人數動輒數十人，再考慮改為子集合 `splits/{id}/members/{memberId}`。

### 3.8 `accounts/{accountId}` — 資產帳戶（淨資產）

| 欄位 | 型別 | 說明 | 範例 |
|---|---|---|---|
| `id` | string | 同文件 ID | `"cash"` |
| `type` | string (enum) | `"cash"` \| `"savings"` \| `"investment"` | `"cash"` |
| `name` | string | 帳戶名稱 | `"現金"` |
| `balance` | number | 餘額 | `28000` |
| `emoji` | string | 圖示 | `"💵"` |
| `color` | string | HEX 色碼 | `"#10A567"` |
| `includeInNetWorth` | boolean | 是否計入淨資產 | `true` |
| `updatedAt` | timestamp | — | — |

**淨資產** ＝ 所有 `includeInNetWorth=true` 帳戶的 `balance` 總和。

> **v1 處理方式**：`cash` 餘額由支出／收入交易自動增減。**投資、儲蓄餘額由 `transfer`（資產搬家）交易驅動**：使用者「記一筆投資／儲蓄」時，系統建立一筆 `transfer`，自動由現金移入對應帳戶，淨資產維持精準（見 2.6、5.2b）。使用者無需手動編輯餘額。

## 4. 安全性規則（Firestore Security Rules）

### 4.1 核心原則

**任何人只能讀寫自己 `uid` 底下的資料，且必須已登入。** 禁止跨使用者讀取，禁止匿名存取。

### 4.2 基本規則範本

```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // 使用者主檔
    match /users/{userId} {
      allow read, write: if request.auth != null
                         && request.auth.uid == userId;

      // 該使用者底下所有子集合（categories / budgets / transactions / goals / splits / accounts）
      match /{document=**} {
        allow read, write: if request.auth != null
                           && request.auth.uid == userId;
      }
    }
  }
}
```

### 4.3 補強建議（上線前）

- **禁止列舉他人**：上述規則已保證讀寫限本人；不要再開任何 collection group 查詢給未授權者。
- **欄位驗證（進階）**：可於規則中加入型別與範圍檢查（例如 `amount` 必須為正數、`type` 必須在允許值內），防止異常寫入。v1 可先靠前端驗證，上線前再補規則層驗證。
- **金鑰限制**：於 Google Cloud Console 對 Web API Key 設定 **HTTP referrer 白名單**（限定你的 GitHub Pages 網域）。

## 5. 關鍵流程與運算邏輯

### 5.1 新使用者初始化（首次 Google 登入）

登入成功且 `users/{uid}` 不存在時，於一個批次寫入（batch）中建立：

1. `users/{uid}` 主檔（含預設 `settings`、空 `badges`）。
2. `categories` 內建分類與收入來源（見 3.3）。
3. `accounts` 三個預設帳戶：現金、儲蓄、投資。
4. 當月 `budgets/{period}` 一份（`income` 可為 0，`allocations` 預設）。

### 5.2 記一筆支出 / 收入（F1）

1. 寫入一筆 `transactions`（帶 `type`、`amount`、`categoryId`、`period`）。
2. 同步調整對應 `accounts` 餘額（支出減現金、收入加現金）。
3. 收入需同步更新當月 `budgets.income`。
4. （選用）更新 `budgets` 的 `spentByCategory` 快取。
5. UI 即時重算該分類使用率，達 `overspendAlertRatio`（預設 85%）跳提醒。

> 步驟 1–3 建議放進**交易型批次（transaction/batch）**，確保「交易紀錄」與「帳戶餘額」一致，不會只寫成功一半。

### 5.2b 記一筆投資 / 儲蓄（資產搬家，F4b）

使用者操作：在投資／儲蓄輸入金額（例：投資 5,000）。系統內部：

1. 寫入一筆 `transactions`，`type="transfer"`、`fromAccountId="cash"`、`toAccountId="invest"`（或 `"savings"`）、`amount=5000`。
2. 於同一批次中：現金帳戶 `−5000`、投資帳戶 `+5000`。
3. **不計入當月收支、不進圓餅圖**；淨資產總額不變，僅內部分布改變。
4. 更新該月投資／儲蓄「目標格」的達成金額（以累計轉入額計）。

> 對使用者而言就是「記一筆投資」，看不到也不需要理解「轉帳」。若現金不足以轉出，前端應提示（可允許負餘額或擋下，見第 7 章已定案：採最後寫入者勝出，不另做鎖定）。

### 5.3 首頁收支圓餅圖（F5）

- 圓餅各扇形 ＝ 當月各分類**支出**交易加總。**投資、儲蓄為 `transfer`（資產搬家），不計入圓餅。**
- 灰色「結餘」扇形 ＝ `income − Σ當月支出 − Σ當月轉入投資/儲蓄`（即真正還躺在現金、尚未花掉也尚未搬走的部分）。
- **圓心顯示「本月結餘」金額**（已確認）。

> 註：因投資／儲蓄改為資產搬家，圓餅的「整體」語意由「收入如何被使用」收斂為「現金的去向（花掉 / 搬去投資儲蓄 / 結餘）」，淨資產因此精準。若希望圓餅仍直觀呈現「存了多少」，可將投資、儲蓄以另一種樣式（例如外圈或不同填色）標示為「已存起來」，與「已花掉」區隔——此為 UI 呈現選項，資料層不受影響。

### 5.4 分帳付款狀態與 7 天提醒（F6 / F8）

- **平分計算**：每人應付 ＝ `floor(totalAmount / 人數)`，**無法整除的餘數一律加到墊款人（payer）身上**。例：3,200 ÷ 3 ＝ 1,066…，則兩人各 1,066，墊款人 1,068（1,066 + 餘 2）。
- 標記某成員已付：讀出 `splits` 文件 → 改該 `members[i].paid = true` 與 `paidAt` → 寫回。
- 全員 `paid=true` 時，設 `status="settled"`、`settledAt`。
- **App 內提醒判斷（每次開啟分帳頁時計算）**：

  ```
  needsReminder =
       status == "open"
    && 任一 member.paid == false
    && (now − createdAt) ≥ settings.splitReminderDays 天
  ```

  命中者於分帳頁標紅 + 首頁顯示提醒徽記。**此為純前端計算，不需後端排程。**

> 主動推播（App 關著也通知，F10）需 Cloud Functions 定時掃描 + FCM，且 iOS 網頁推播需使用者先「加入主畫面」。列為 v2+ 進階，不阻擋 v1。

### 5.5 餘額校正 / 對帳（F9，v2 規劃）

1. 使用者輸入某帳戶的「實際餘額」。
2. 系統計算差額 ＝ 實際餘額 − 系統記錄餘額。
3. 自動建立一筆 `adjustment` 交易：
   - 差額為負（錢變少）→ 分類掛「不明損失」。
   - 差額為正（錢變多）→ 分類掛「不明收入」（屆時新增此收入來源）。
4. 更新帳戶餘額至實際值。

> 此機制讓「存摺對不上的零頭」有系統化的歸帳方式，比每次用猜的手動記更準。v1 先提供「不明損失」分類供手動記，F9 再補自動對帳入口（規劃放在資產卡）。

### 5.6 徽章解鎖（F3 周邊）

前端常數表定義每個徽章的解鎖條件（例：首次存錢、投資啟動、連續記帳 7 天、完成目標、淨資產破 10 萬、零超支月）。事件發生時檢查條件，未解鎖則寫入 `users/{uid}.badges.{id}`。

## 6. 開發階段拆解與驗收標準

> 建議順序，每階段可獨立交付、獨立驗收。前一階段穩定再進下一階段。

### 階段 0 — 互動原型（已完成）
- 現狀：單檔 `index.html`，資料寫死於前端記憶體，重整即重置。
- 用途：UI 驗證、朋友試玩、收集回饋。

### 階段 1 — 本機持久化（localStorage，選用過渡）
- **目標**：讓試玩資料重整後仍在，免後端即可收集更真實回饋。
- **工作項**：將前端 `state` 物件序列化存入 `localStorage`，啟動時載入。
- **驗收**：新增交易 / 拉預算 / 建目標後重整頁面，資料保留。
- **備註**：此為過渡，資料僅存於該瀏覽器，換裝置不會跟著走。可視回饋進度決定是否略過、直接進階段 2。

### 階段 2 — Firebase 專案與 Google 登入
- **工作項**：
  1. 建立 Firebase 專案、啟用 Authentication（Google 提供者）。
  2. 設定 OAuth 同意畫面、授權網域（含 GitHub Pages 網域）。
  3. 前端引入 Firebase SDK（CDN），實作登入 / 登出 / 登入狀態監聽。
  4. 未登入導向登入頁，登入後進入 App。
- **驗收**：可用 Google 帳號登入，重整後維持登入狀態，登出正常。
- **參考**：可沿用既有背單字 App 的 Google 登入經驗與設定流程。

### 階段 3 — Firestore 串接（核心）
- **工作項**：
  1. 套用第 4 章安全規則。
  2. 新使用者初始化流程（見 5.1）。
  3. 將原型所有讀寫改接 Firestore CRUD：交易、分類、預算、目標、分帳、帳戶。
  4. 以 `onSnapshot` 即時監聽，資料變動即時反映於 UI。
  5. 啟用 Firestore 離線持久化（offline persistence），改善行動網路不穩體驗。
- **驗收**：
  - A 手機登入記一筆 → B 手機（同帳號）登入可見同一筆。
  - 換手機重新登入，所有資料完整還原。
  - 安全規則測試：嘗試讀取他人 uid 資料應被拒絕。

### 階段 4 — 依賴資料的 App 內邏輯
- **工作項**：投資／儲蓄資產搬家（`transfer`，5.2b）與淨資產精準計算、分帳 7 天提醒（5.4）、徽章解鎖（5.6）、首頁圓餅彙總（5.3）、超支提醒（5.2）。
- **驗收**：
  - 記一筆投資 5,000 → 現金 −5,000、投資 +5,000、淨資產總額不變、該筆不出現在收支圓餅。
  - 建立一筆 8 天前的未結清分帳 → 分帳頁標紅、首頁出現提醒。

### 階段 5 — 進階（later）
- 餘額校正 / 對帳（F9）。
- 主動推播（F10，Cloud Functions + FCM）。
- 資料匯出（CSV）。
- 分帳跨帳號連動（F11，需重新設計共享資料結構與權限）。
